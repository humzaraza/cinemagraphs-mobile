Cinemagraphs PR 1b — Banner Picker

BACKDROP and PHOTO modes (Gradient mode shipped in PR 1a)
Mockup v3 — May 6 2026

For Claude Code: Match these mockups exactly when building the picker UI. Existing screen is app/header-picker.tsx (currently shipping gradient mode only). PR 1b extends it with BACKDROP and PHOTO modes via a segmented control above the mode-specific content area.

Critical aspect ratio rule: The preview area MUST be exact 16:9 (aspect-ratio: 16/9 in CSS, aspectRatio: 16/9 in React Native style). This matches the actual profile banner aspect, so what the user sees in the picker preview is exactly what they get on their profile (WYSIWYG). Do not approximate with a fixed height.

Key rules: No em dashes. Gold #C8A951, Teal #2DD4A8, Ivory #F5F0E1, Background #0D0D1A. DM Sans only (Playfair dropped May 3 2026). Preview area updates live as user picks.
Locked decisions:
• bannerValue for BACKDROP = TMDB filmId (string). Resolved to backdrop URL at render via getBannerUrl helper.
• bannerValue for PHOTO = Vercel Blob path. Resolved to full CDN URL at render.
• Photo crop ships with full position adjuster in v1. Pan-to-reposition gesture. ~200-300 lines via expo-image-picker + expo-image-manipulator + react-native-gesture-handler. Centered-only is the recovery fallback if CC stalls on the gesture handler — adjuster becomes a polish PR (1c) in that case.
• Popular films default = catalog popularity sort. Server query /api/films/search?sort=popular&limit=12 when q is empty. Endpoint branches on q presence: empty q returns popularity slice, non-empty q returns ILIKE/full-text title match. All results filter backdropPath IS NOT NULL.
• Search results = grid layout (consistent with Popular Films, not list).
• Tab-switch resets preview to whatever's currently persisted (until user makes new selection in new tab).
• Save button shows "Saving..." with spinner during PHOTO upload. Disabled state. Navigate back on success, alert+stay on failure (matches PR 1a-followup pattern). GRADIENT and BACKDROP saves are quick PATCHes with no spinner.
• Vercel Blob upload uses handleUpload from @vercel/blob/client for client-to-blob direct upload with server-side token authorization. Avoids the 4.5MB Vercel function body limit.
• Backdrop image source = TMDB backdrop_path (clean cinematography stills, no text).
• Segmented labels = "Gradient / Backdrop / Photo".
• PR 1b is mobile + web companion (web work: PATCH extension + filmId-exists validation + Vercel Blob upload route + films search endpoint + old-blob cleanup logic).
• Old blob cleanup fires on ANY bannerType transition away from PHOTO (not just PHOTO-to-PHOTO). Logic in PATCH: if previous bannerType was PHOTO and previous bannerValue is non-null, attempt blob delete before writing new bannerType/Value. Try-catch around delete with log-and-continue on failure (don't fail the PATCH).

Known limitations / parked followups

Issues acknowledged but not blocking PR 1b. Captured here so future-Claude and future-CC don't surface them as if they're new.

1. Mid-upload abandonment can leak orphan blobs. If a user starts a Photo upload, the blob upload completes successfully, but then the PATCH to /api/user/banner never reaches the server (network drop after upload, app crash, force-quit), the blob is stranded with no DB reference. PATCH idempotency handles the case where PATCH lands but client doesn't see the response (next save replaces cleanly). This leaky-blob case requires a future periodic sweep cron — not blocking v1.

2. Account deletion does not clean up photo banner blob. When a user deletes their account, their photo banner Vercel Blob asset becomes orphaned unless account deletion explicitly cleans it up. Out of scope for PR 1b (account deletion lives in Settings PR), but flagging here so it's not forgotten.

3. Centered-crop fallback for Photo adjuster. If the pan-to-reposition gesture handler stalls in CC across multiple sessions, the recovery move is to ship Photo mode with centered-only crop and file the adjuster as a polish PR (1c). This is a backstop, not the goal. Goal is full adjuster in v1.

4. TMDB backdrops with text/logos. Most TMDB backdrop_path images are clean cinematography stills, but a small percentage include title cards, promotional text, or studio logos. These look bad as banners. v1 does not filter for this; if a film looks bad as a banner, the user notices and picks a different film. If this becomes a recurring complaint, add a manual override system (per-film hand-curated backdrop) in a later PR.
